# PDF Dictionary App

This is a sample iOS application written in **SwiftUI** that demonstrates how to build
a PDF reader with file-management capabilities and an integrated dictionary lookup
feature.  The app lets you import PDF documents, organise them into folders,
read them with rich zooming and navigation, and look up word definitions on the
fly.

## Features

* **PDF management:** Browse and organise PDF files stored in the app’s Documents
  directory.  Create new folders to group related books or papers and import new
  PDFs from the system’s Files app.  The file browser persists your library
  between launches because it uses the standard sandboxed Documents directory.

* **PDF viewing:** Display PDF documents using Apple’s PDFKit.  Readers can
  scroll, zoom and navigate through pages.  PDFKit handles the heavy lifting
  of rendering, searching and annotating documents.  According to Apple’s
  documentation, PDFKit provides built-in rendering and navigation, supports
  searching within a document and allows developers to extract text and add
  annotations such as highlights or shapes.  These
  capabilities are leveraged in this app.

* **Dictionary look up:** A toolbar button toggles a “dictionary mode.”  When
  enabled, selecting a single word in the PDF causes a definition to appear in
  a side panel (or sheet on narrow screens).  iOS exposes access to its
  built-in dictionaries only through `UIReferenceLibraryViewController` — a
  view controller that displays definitions for a term.  The app embeds this view controller in SwiftUI to present definitions
  inline.

* **Universal design:** Thanks to SwiftUI, the app adapts gracefully to
  different devices.  On an iPad, the library, document viewer and dictionary
  panel appear side by side in a multi-column layout.  On an iPhone, the
  dictionary slides up as a sheet.

## Building and running

To try the app, follow these steps:

1. Open **Xcode** (version 14 or later) on your Mac.
2. Choose *File → New → Project*, select the **iOS App** template and click
   **Next**.  Give the project a name (for example, *PDFDictionaryApp*) and
   choose **SwiftUI** as the interface and **Swift** as the language.
3. After Xcode creates the project, delete the autogenerated
   `ContentView.swift` and `<YourAppName>App.swift` files from the
   navigator.
4. Drag the files from the `pdf_dictionary_app` folder (contained in this
   archive) into your project’s navigator.  Make sure “Copy items if
   needed” is checked.
5. Add the **PDFKit** framework if it isn’t already included.  PDFKit has been
   part of the iOS SDK since iOS 11 and allows apps to display and search
   PDF documents.
6. Build and run on an iPhone or iPad simulator (or a physical device).

### Creating an installable `.ipa` file

To distribute the app outside of Xcode (for example, to beta testers), you need
to generate an **IPA**.  This process requires membership in the Apple
Developer Program because you must create a distribution certificate, an
Application ID, register any test devices and generate a provisioning profile.
DeployGate’s documentation summarises the prerequisites for archiving and
exporting an iOS app: you must select a real device (not a simulator) as the
build target and create a distribution certificate, App ID, registered test
devices and a provisioning profile.

Once you have configured signing in Xcode, you can create an IPA from the
command line or through the Xcode UI:

1. **Archive the app:** In Xcode, choose *Product → Archive*.  When the build
   succeeds, the Archive organizer will open.
2. **Export:** Click *Distribute App*, choose the distribution method (Ad Hoc,
   Development or App Store), select the appropriate provisioning profile and
   export.  Xcode produces an `.ipa` file.

Alternatively, the `build_ipa.sh` script included in this project shows how to
use `xcodebuild` to archive and export the app programmatically.  You must
create an `ExportOptions.plist` to specify the signing method and provisioning
profile.  Running the script will create a `.xcarchive` and an IPA in the
`build/ipa` directory.

No external dependencies are required; everything uses built-in system
frameworks.  The code is fully contained in the Swift files provided.
